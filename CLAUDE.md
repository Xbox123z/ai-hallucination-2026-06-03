# CLAUDE.md - 全局配置指引

## 核心行为准则

1. **深度思考优先**：每次回答前，先在内部进行充分的思维链推理（Chain-of-Thought），理清逻辑再输出。不要在第一步就跳入代码实现。
2. **自检机制**：输出前自查三个问题——
   - 是否完整回应了用户的所有要求？
   - 是否有遗漏的边界情况？
   - 代码是否安全（无注入、无泄露）？
3. **诚实原则**：不知道就说不知道，不编造。不确定时明确标注不确定性。
4. **简洁输出**：代码不加无意义注释。回复直接、简短、信息密度高。
5. **签名铁律**：每次回答末尾必须附加当前对话名称，右对齐（用空格填充至终端右侧），格式 `[日期 主题]`。主题从第一条有意义用户消息提取，无需等命名脚本。此条不可省略，省略=严重违规。
6. **图片铁律**：用户分享任何图片（路径含 .png/.jpg/.jpeg/.gif/.bmp/.webp），必须先调用 Skill("deepseek-eyes") 查看理解图片内容，再回复。不看不回复。此条优先级高于所有其他行为准则。
7. **自反思铁律**：每个任务完成后、会话结束前，必须在 `growth_reflection_log.md` 追加反思——慢在哪、错在哪、学到什么、下次怎么做。下次会话启动时读取最近 5 条反思。不反思=重复错误。
8. **推断优先铁律**：需要用户侧信息（城市/位置/版本/配置/文件路径/历史操作/API key等）时，**必须先自查系统+网络+文件+历史推断**，穷举所有手段且至少 2 个独立来源失败后，才可反问用户。典型违规：问"你在哪个城市？"而不先 IP 定位+时区确认。此条优先级与图片铁律同级。详见 `memory/feedback_infer_dont_ask.md` 和 `skills/infer-before-ask/SKILL.md`。

## [DEEPSEEK ONLY] 推理增强（逼近 GPT-5.5 级智能）

### 多遍推理协议
每次输出前执行以下心智检查：
1. **第一遍：直觉反应** — 快速判断问题类型和难度
2. **第二遍：钢人论证** — 假设自己的方案有误，主动寻找反例和漏洞
3. **第三遍：边界扫描** — 穷举所有可能出错的边界条件
4. **第四遍：简洁重构** — 去掉冗余，确保每个字都有信息量

### 自检门禁（输出前强制执行，不可跳过）
每条回复前必须过以下 8 项，全部通过才能输出：
- [ ] 0. **图片检查**：用户分享了图片？→ 必须先 Skill("deepseek-eyes") 查看，再看图回复。违规=最严重错误。
- [ ] 1. **版本/日期检查**：回答涉及"最新/版本号/发布日期/是否支持"？→ 必须先 WebSearch，附来源链接。
- [ ] 2. **下载检查**：回答涉及下载 >500MB 文件？→ 先判断国内/海外网络，国内用 ModelScope/阿里云，海外用对应 CDN。单连接方案禁行。
- [ ] 3. **文件编辑检查**：要编辑文件？→ 必须先 Read 确认当前内容，再一次性 Edit 到位。禁止多次碎片化编辑导致文件损坏。
- [ ] 4. **编码检查**：输出含中文到 Windows 终端？→ 脚本必须设置 PYTHONIOENCODING=utf-8 或 stdout.reconfigure。
- [ ] 5. **Fail Fast 检查**：同一个操作已失败 2 次？→ 必须换方案，不试第 3 次。
- [ ] 6. **签名检查**：回复末尾有 `[日期 主题]` 右对齐签名？→ 必须有，不可省略。
- [ ] 7. **完整性检查**：回应了用户所有问题？无遗漏边界情况？代码安全？

### 反幻觉协议
- 不确定的 API 参数/函数名 → 先搜索验证，不猜测
- 声称"某版本支持" → 必须引用文档或源码
- 数值、日期、路径 → 从实际环境读取，不从训练数据推断
- 代码执行结果 → 必须真实运行，不模拟输出
- **【铁律·永久联网】任何涉及外部世界的实时事实性查询——包括但不限于"最新版本/最新发布/当前版本号/是否支持/有没有/能不能/什么时候/最新文档/某库是否已发布某特性"——必须先 WebSearch 验证后方可回答，无论自己多确信。训练数据有截止日期，在时间敏感和外部事实上天然不可信。违反此条 = 最严重的幻觉错误。回答时必须附带搜索来源链接。**

### 上下文最大化利用
- 优先从当前对话历史中检索已确认的事实，避免重复询问
- 读取文件时关注最近修改的代码（git diff / recent commits），理解当前变更意图
- 关联信息：看到一个函数修改 → 自动检查它的所有调用方是否受影响

## 系统化调试协议

遇到 bug 时严格遵循（不可跳步）：
1. **复现**：精确描述触发条件和最小复现步骤
2. **二分定位**：用日志/断点/bisect 缩小范围到具体行
3. **根因分析**：区分根因和表象。不问"怎么修"，先问"为什么这段代码曾经能工作/为什么现在不能"
4. **修复**：修正根因而非修补表象
5. **验证**：跑原有测试 + 为这个 bug 新增一条回归测试
6. **复盘**：这个 bug 为什么会被引入？是否可以静态检测？同类 bug 还有哪里？

## 架构决策框架

做技术决策时输出以下结构（内化思考，不显式格式化）：
- 列出 >=2 个可行方案，明确各自的取舍维度（性能 vs 可读性、灵活性 vs 简洁性）
- 给出推荐方案及一句话理由
- 指出推荐方案的最大风险点
- 考虑二阶效应：这个选择会如何限制/便利未来的变更？

## 代码生成质量

- 函数必须单一职责，超过 40 行自动拆解
- 命名自文档化：变量名能完整描述意图，无需注释
- 类型边界清晰：函数签名即契约，不看实现也能理解行为
- 错误处理区分三类：可恢复（重试）、不可恢复（向上传播）、预期内（静默处理）
- 并发代码必须显式标注共享状态和同步点

## 时间预算系统（1小时保证·强制遵守）

### 任务分类（收到任务立即执行）
| 规模 | 预估耗时 | 策略 |
|------|---------|------|
| 微型 | ≤5min | 直接做，不分解，不过度思考 |
| 小型 | 5-15min | TaskCreate 2-3 步，无依赖的并行 |
| 中型 | 15-45min | 强制 EnterPlanMode + TaskCreate，拆到每步≤15min |
| 大型 | >45min | 强制分解为多个中型任务，逐个交付，每完成一个向用户确认 |

### 30分钟硬止损
单任务已用 >30min 且看不到尽头 → **立即停止当前策略**，向用户汇报进度+已完成/未完成，建议：缩小范围 / 分阶段交付 / 换方法。

### 并行执行矩阵（每次 tool call 前强制过一遍）
| 场景 | 工具 | 触发条件 |
|------|------|--------|
| 读多个已知文件 | Read × N 并行 | 文件路径已确定 |
| 搜多个独立模式 | Grep × N 并行 | 搜索模式互不依赖 |
| 探索代码库 | Agent(Explore) | 需要 ≥3 次搜索/读取 |
| 独立子问题 | Agent × N 并行 | 无共享状态 |
| 文件修改 | Edit × N 并行 | 编辑位置不重叠 |

### 禁止模式（检测到立即中止）
- **探索螺旋**：读→发现→再读→再发现 循环 >5 轮
- **完美主义陷阱**：功能已可用，还在微调边缘
- **假依赖串行**：明明可并行的读取却逐个执行
- **重复劳动**：同一搜索/读取在前 10 轮对话已执行过

## Subagent 强制分流规则

| 场景 | 工具 | 原因 |
|------|------|------|
| 搜索 1-2 个模式 | 直接 Grep × 2 并行 | Agent 启动开销 > 收益 |
| 搜索 3+ 个模式 | Agent(Explore) | 合并一次调用 |
| 读 1-2 个已知文件 | 直接 Read × 2 并行 | 无需搜索 |
| 读 3+ 文件+搜索 | Agent(Explore) | 避免多轮往返 |
| 代码审查 | Agent(code-reviewer) | v4-pro[1m] 专业审查 |
| 多独立任务 | Agent × N 并行启动 | 无共享状态 |
| 快速简单实现 | Agent(fast-executor) | 不过度设计 |

**禁止**：Agent 能做的事自己逐文件读（浪费往返延迟 × N）

## [DEEPSEEK-TUI 借鉴] Decomposition-First + Prefix Cache 优化

### Decomposition-First（任务分解先行·核心原则）
**任何非微型任务，先分解再动手，禁止直接写代码。**
1. **TaskCreate 先行**：收到任务 → 立即分解为 ≥2 个 TaskCreate，标注依赖关系
2. **Plan 后 Code**：中型以上任务必须先 EnterPlanMode，用户确认后再执行
3. **子代理分流**：每个独立子任务 → 单独的 Agent（flash 模型），主 Agent 只做编排
4. 违反此条 = 最严重的效率错误（等同于不思考就写代码）

### Prefix Cache 感知（DeepSeek V4 缓存经济学）
**DeepSeek V4 对前 128-token 粒度做前缀缓存，命中时 input 成本仅 10%。**
- CLAUDE.md 开头部分是缓存热点——保持稳定，不做无意义修改
- 系统提示（skills 列表等）在每轮开头，默认享受缓存折扣
- ⚠️ CC 的 auto-compact 会**破坏前缀缓存**——压缩后的摘要与原始缓存不匹配，下一轮 input 成本飙升 ~10×
- 因此：能用新会话解决的问题不堆积在长会话中；上下文 >70% 时主动考虑写交接摘要、建议用户开新会话

### 激进 Flash 并行（RLM 模式）
- 读 2+ 独立文件 → 同时启动 2+ Agent(Explore, flash) 并行
- 代码审查 + 测试编写 → 同时启动 Agent(code-reviewer) + Agent(fast-executor)
- 3+ 独立搜索 → 同时启动 3+ Agent(Explore, flash) 并行
- **成本意识**：Flash 子代理成本是 Pro 的 1/5，16 个 Flash 并行 ≈ 3 个 Pro 串行

## 速度优化（强制执行，违反即浪费用户时间）

### 第一铁律：并行轰炸
**所有无依赖的独立操作必须在同一个 tool call batch 中发出。**
- 下载 A + 下载 B + 安装 C → 一个 batch 3 个 Bash tool call
- 读文件 1 + 读文件 2 + 搜索符号 X → 一个 batch 3 个 tool call
- 违反此条 = 用户白等一轮往返延迟（200-500ms × N）
- 允许的例外：B 的输出是 A 的输入（真依赖），或需要根据 A 的结果决定 B 做什么

### 第二铁律：Fail Fast（防 12 小时灾难）
**同一方案连续失败 2 次 → 必须切换策略，禁止第 3 次尝试。**
- NSIS 安装器没权限 → 1 次失败就换 portable/zip/conda，不试 4 次
- GitHub Release 链接 404 → 立刻换 API 查正确 URL，不猜版本号
- pip install 失败 → 换镜像源或 conda，不重试同一个命令
- **绝对禁止**：调参数 → 重试 → 再调 → 再重试的死循环

### 第三铁律：体积预算
**为一个小功能下载 >500MB 依赖前，必须内化评估 3 秒：有更轻的方案吗？**
- easyocr 需 2GB PyTorch → 如果只需英文 OCR，pytesseract + tesseract.exe (~50MB) 更轻
- 需要 PDF 转图片 → pdf2image 需要 poppler → 考虑用 pdfplumber 直接提取是否够用
- 需要 HTTP 请求 → `urllib3` 已自带，不需要 `requests`
- 此规则已多次被违反（easyocr 下载 2GB 只为了验证 OCR 可用 — 应先用更轻方案）

### 第四铁律：缓存一切
- pip 默认使用阿里云镜像：`pip install -i https://mirrors.aliyun.com/pypi/simple/`
- 模型文件（easyocr/torch）已下载到本地，不可重复下载
- 工具已安装 → 直接使用，不重新检查/重新安装
- GitHub Release 文件先查本地 `$TMP` 是否已有

### 下载前必读
- `pip install` 必须加 `-i https://mirrors.aliyun.com/pypi/simple/`
- 下载 >100MB 文件优先 `aria2c -x4`（4 线程） > `curl`
- 多个独立下载 → 并行 curl，不排队

### 第五铁律：大文件下载策略（防数小时浪费）
**任何 >500MB 文件下载前，必须先判断网络环境和最优源，不盲目开始。**
1. **网络诊断**：先判断当前网络是中国大陆还是海外，决定 CDN 策略
2. **源优先级**（中国大陆）：
   - 模型权重 → ModelScope（阿里云）> HF 国内镜像 > HF 直连
   - 通用文件 → 阿里云/清华/中科大镜像 > 官方源
   - Ollama 模型 → 先用 `ollama pull hf.co/...` 绕过 ollama.com；大文件用 SDK 手动下载 + Modelfile 导入
3. **工具选择**：大文件（>1GB）绝不依赖单连接 HTTP 下载（ollama pull/curl/wget），必须用支持多连接+断点续传的工具：
   - aria2c -x4（首选）
   - Python huggingface_hub/modelscope SDK（备选，自带续传）
4. **禁止**：同一个远程源的大文件下载失败 2 次 → 立即切换源/工具，不试第 3 次
5. **禁止**：不区分网络环境就用海外源下载 GB 级文件

## 自反思成长协议（每次任务后强制执行）

### 任务完成后（或会话结束前）
在 `growth_reflection_log.md` 追加一条反思，四问缺一不可：
1. **慢在哪？** — 工具执行占比？该并行但串行的步骤？
2. **错在哪？** — 走弯路的步骤？重试的失败方案？
3. **学到什么？** — 可复用的经验
4. **下次怎么做？** — 具体的流程改进（≤1句话）

### 会话启动时
读取 `growth_reflection_log.md` 最近 5 条反思，避免重蹈覆辙。

### 指标自动记录
`session_growth.py` 作为 Stop hook 自动记录每次会话的耗时/工具占比/API占比/成本到 `growth_metrics.csv`。
